# 🧩 Архитектура проекта на Prism (WPF + Unity)

В определённый момент я пришёл к тому, что держать всю логику лаунчера в `MainWindow` — это тупиковый путь.
Когда проект начинает расти, в одном окне начинает жить вообще всё подряд: запуск игры, обновления, настройки, трей, работа с конфигами, сервисы, UI-состояния и куча обработчиков.

Поэтому я перевёл структуру проекта на **Prism + Unity**, чтобы разделить ответственность, сделать код чище и подготовить проект к дальнейшему расширению.

# ⚙️ Почему я выбрал Prism + Unity

Я использую **Prism** как основу архитектуры WPF-приложения, а **Unity Container** — для управления зависимостями.

Это даёт мне несколько важных преимуществ:

- я **не завязываю весь проект на один MainWindow**
- могу **разносить функциональность по модулям**
- могу **нормально внедрять зависимости**
- логика UI уходит во **ViewModel**, а не размазывается по code-behind
- приложение проще масштабировать, поддерживать и тестировать

Если коротко — это уже не “окно с кнопками”, а **нормально организованное клиентское приложение**.

# 🧠 Что для меня поменялось после перехода

Раньше у меня многое жило прямо внутри `MainWindow.xaml.cs`:

- запуск клиента
- обновление файлов
- работа с конфигами
- открытие окон
- трей
- обработка состояния приложения

С переходом на Prism я начал раскладывать это по назначению:

## Что я выношу в `Services/`

Сюда я складываю всю прикладную логику, которая **не должна жить в UI**.

Например:

- запуск игры
- работа с путями
- чтение/сохранение настроек
- обновление клиента
- проверка состояния файлов
- логика лаунчера

Пример:

```csharp
public interface IGameService
{
    void LaunchGame();
}
```

```csharp
public class GameService : IGameService
{
    public void LaunchGame()
    {
        // логика запуска игры
    }
}
```

---

## Что я выношу в `ViewModels/`

Во ViewModel я держу **состояние окна и команды**, а не саму бизнес-логику.

Например:

- текст кнопок
- флаги состояния
- выбранные параметры
- команды запуска / обновления / открытия окон

Пример:

```csharp
public class MainWindowViewModel : BindableBase
{
    private readonly IGameService _gameService;

    public DelegateCommand LaunchGameCommand { get; }

    public MainWindowViewModel(IGameService gameService)
    {
        _gameService = gameService;
        LaunchGameCommand = new DelegateCommand(OnLaunchGame);
    }

    private void OnLaunchGame()
    {
        _gameService.LaunchGame();
    }
}
```

То есть `ViewModel` говорит:
**“что делать”**, а `Service` уже знает:
**“как именно это делать”**.

---

## Что я оставляю во `Views/`

Во `Views` у меня лежит только визуальная часть:

- XAML
- разметка
- привязки (`Binding`)
- визуальные контейнеры
- регионы Prism

То есть окно должно **отображать**, а не “думать за весь проект”.

---

## Что я выношу в `Modules/`

Когда функциональность становится крупной, я не держу её в общей куче, а выделяю в отдельный модуль.

Например:

- модуль настроек
- модуль обновлений
- модуль управления игрой
- модуль диагностики
- модуль новостей / контента

Это позволяет не превращать проект в свалку из 200 файлов в одном месте.

# 🔄 Как теперь работает жизненный цикл приложения

С Prism запуск приложения выглядит уже более правильно и предсказуемо.

## Последовательность такая:

1. Запускается `App.xaml.cs`
2. Приложение наследуется от `PrismApplication`
3. Prism вызывает `RegisterTypes()`
4. Я регистрирую сервисы и зависимости
5. Prism вызывает `CreateShell()`
6. Создаётся главное окно (`MainWindow`)
7. Загружаются модули
8. В нужные регионы подгружаются представления

---

# 🏗 Что важно в Prism + Unity

## 1. `App.xaml.cs` наследуется от `Prism.Unity.PrismApplication`

То есть приложение запускается уже не как обычный WPF App, а как **Prism Shell-приложение**.

Пример:

```csharp
public partial class App : PrismApplication
{
    protected override Window CreateShell()
    {
        return Container.Resolve<MainWindow>();
    }

    protected override void RegisterTypes(IContainerRegistry containerRegistry)
    {
        containerRegistry.RegisterSingleton<IGameService, GameService>();
        containerRegistry.RegisterSingleton<ISettingsService, SettingsService>();
    }
}
```

---

## 2. Все зависимости я регистрирую централизованно

Вместо ручного `new SomeService()` по всему проекту я регистрирую всё через контейнер:

```csharp
containerRegistry.RegisterSingleton<IGameService, GameService>();
containerRegistry.RegisterSingleton<ISettingsService, SettingsService>();
```

Это удобно, потому что потом сервисы просто **внедряются автоматически** туда, где они нужны.

---

## 3. Главное окно не создаётся вручную

Это важный момент.

Раньше можно было где-то написать:

```csharp
var window = new MainWindow();
window.Show();
```

В Prism я этого не делаю.
`MainWindow` создаётся как **Shell** через контейнер зависимостей.

То есть приложение строится уже как **архитектура**, а не как набор ручных вызовов.

---

## 4. Логика уходит из окна во ViewModel

Я стараюсь не держать в `MainWindow.xaml.cs` ничего лишнего, кроме реально UI-специфичных вещей.

То есть если действие:

- запускает игру
- проверяет конфиг
- обновляет данные
- работает с файлами

то это уже не задача окна.

---

# 🧱 Что я считаю правильным разделением в проекте

Если коротко, я для себя разделяю проект так:

## `Services`

> Всё, что делает работу

## `ViewModels`

> Всё, что управляет состоянием и командами UI

## `Views`

> Всё, что отображается пользователю

## `Models`

> Всё, что хранит данные

## `Modules`

> Всё, что можно логически выделить в отдельную функциональность

---

# 🚫 Почему я ушёл от “всё в MainWindow”

Потому что это очень быстро начинает ломаться по всем фронтам.

Когда в одном окне лежит всё подряд:

- становится тяжело читать код
- сложно что-то переиспользовать
- сложно добавлять новые функции
- трудно отлаживать
- проект начинает “цементироваться”

А Prism даёт мне возможность строить приложение **нормально, как систему**, а не как один огромный обработчик кнопок.

# ✅ Итог

Переход на **Prism + Unity** для меня — это не “мода на паттерны”, а способ держать проект в адекватном состоянии по мере роста.

Я использую эту архитектуру, чтобы:

- разделять ответственность
- не захламлять `MainWindow`
- удобно расширять функциональность
- подключать модули
- держать зависимости под контролем
- строить лаунчер как полноценное приложение, а не временный UI-скрипт
